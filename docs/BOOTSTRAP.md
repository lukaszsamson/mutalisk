# Mutalisk Bootstrap & Overlay Design

This document is the design contract for how mutalisk loads itself into a target Mix project's child processes (oracle compile, schema compile, fallback recompile, worker `mix test`). It is the M0 starter for milestones M2, M7, M8, and M10.

It implements the **Child-Process Bootstrap Contract** from `PLAN.md` §"Operational Contracts".

## Problem statement

Compiler tracer options and module loading do not cross OS process boundaries. Mutalisk runs in the host BEAM but drives a child Mix process to compile the target. The child has its own code path, dep set, application supervision tree, and `Code` compiler options — all separate from the host. To get our tracer registered, our runtime started, and our env-var-driven mutant selector active inside the child, we must inject mutalisk **into the target's `mix.exs`** for the duration of a run.

We never modify the user's source tree. All injection happens in a working copy.

## Working-copy layout

A working copy is created per run by `Mut.WorkCopy.materialize/2`:

```
tmp/mut_work/<run_id>/                 # one per host invocation
├── mix.exs                             # WRITTEN: overlay
├── mix_user.exs                        # MOVED FROM the user's mix.exs verbatim
├── mix.lock                            # SYMLINK to user's mix.lock
├── lib/                                # COPIED (COW) from user; mutated for schema role
├── test/                               # COPIED (COW) from user
├── config/                             # SYMLINK
├── deps/                               # SYMLINK to user's deps
├── priv/                               # SYMLINK if present
└── _build/
    ├── mut_oracle/                     # populated by oracle role
    └── mut_schema/                     # populated by schema role
```

For workers, `Mut.Sandbox.create_pool/2` clones the working copy into per-worker dirs at `tmp/mut_sandboxes/<run_id>/<n>/`. Workers reuse the same `mix.exs`, `lib/`, `test/`, `_build/mut_schema` as the schema build.

**Umbrella projects are out of scope for v1.** Detect via `apps_path` in the user's `project/0` and fail loudly with a clear "umbrella not supported in v1" error.

## Roles

A single env var, `MUTALISK_ROLE`, governs overlay behavior. Set by the host on every child Mix invocation.

| Role | Set by | Used by | Effect on overlay |
|---|---|---|---|
| `oracle` | `Mut.OracleBuild` | M2 | Prepends `:mut_oracle` (resolves to `Mix.Tasks.Compile.MutOracle`) to `:compilers` |
| `schema` | `Mut.SchemaBuild` | M7 | Plain wrapper; relies on dep-graph ordering for runtime boot |
| `worker` | `Mut.Worker.run_schema` | M8 | Same as `schema`; only `mix test --no-compile` runs |
| `fallback` | `Mut.Worker.run_fallback` | M10 | Same as `schema`; `mix compile` then `mix test --no-compile` |

The host also sets:

- `MUTALISK_PATH` — absolute path to the host mutalisk checkout, used as the `:path` dep target in the overlay.
- `MUT_ACTIVE` — integer mutant id (worker/fallback only). Read by `Mut.Application.start/2`.
- `MIX_ENV=test` and `MIX_BUILD_PATH=<role-specific>` — per the Build-Path Contract.

## Overlay `mix.exs` (concrete template)

The overlay file written into the working copy. It is fully self-contained — no runtime dependency on the host mutalisk modules, because mutalisk is a **`:path` dep loaded by Mix, not by the overlay**.

```elixir
# tmp/mut_work/<run_id>/mix.exs (generated by Mut.Bootstrap.Overlay)
Code.require_file("mix_user.exs", __DIR__)
mutalisk_user_mod = Mix.Project.get!()
Application.put_env(:mutalisk, :user_mix_module, mutalisk_user_mod)

defmodule Mutalisk.WrappedMixProject do
  use Mix.Project

  @mutalisk_path System.fetch_env!("MUTALISK_PATH")
  @mutalisk_role System.get_env("MUTALISK_ROLE", "schema")

  def project do
    user_mod = Application.fetch_env!(:mutalisk, :user_mix_module)
    user = user_mod.project()

    user
    |> Keyword.update(:deps, default_deps(), &add_mutalisk_dep/1)
    |> Keyword.update(:compilers, default_compilers(), &maybe_prepend_oracle_compiler/1)
  end

  def application do
    user_mod = Application.fetch_env!(:mutalisk, :user_mix_module)

    if function_exported?(user_mod, :application, 0) do
      user_mod.application()
    else
      []
    end
  end

  defp default_deps, do: []
  defp default_compilers, do: Mix.compilers()

  defp add_mutalisk_dep(deps) do
    if Enum.any?(deps, fn
         {:mutalisk, _} -> true
         {:mutalisk, _, _} -> true
         _ -> false
       end) do
      deps
    else
      [{:mutalisk, path: @mutalisk_path, only: [:test], runtime: true} | deps]
    end
  end

  defp maybe_prepend_oracle_compiler(compilers) do
    case @mutalisk_role do
      "oracle" -> [:mut_oracle | compilers]
      _ -> compilers
    end
  end
end
```

The `:mut_oracle` atom is resolved by Mix to `Mix.Tasks.Compile.MutOracle` via Mix's standard `Mix.Tasks.Compile.<Camelize>` lookup. The module is shipped in mutalisk and loaded into the child BEAM via the path-dep mechanism. Mix's convention is **atoms in `:compilers`, not `{atom, module}` tuples** — `{:mut_oracle, SomeOtherModule}` would not work and would silently misbehave.

### Why this works

- `Code.require_file("mix_user.exs", ...)` evaluates the user's original mix.exs. Their `defmodule MyApp.MixProject do use Mix.Project end` runs, and `use Mix.Project` calls `Mix.Project.push(MyApp.MixProject)`, registering it.
- `Mix.Project.get!()` returns the just-registered user module. We stash it via `Application.put_env/3` so `Mutalisk.WrappedMixProject` can fetch it lazily (we cannot capture it in a module attribute because module-attr expressions evaluate at compile time of *each function*, not at file-eval time, and the local var isn't in scope inside `defmodule`).
- `defmodule Mutalisk.WrappedMixProject do use Mix.Project end` then pushes our wrapper. Mix dispatches to the most recently pushed project, so our `project/0` runs.
- We delegate to the user's `project/0` and apply two overrides: dep injection (mutalisk path-dep) and compiler injection (oracle compiler when role demands).
- We delegate `application/0` unchanged. Mutalisk's app starts before the user's because OTP derives boot order from the dep graph, and mutalisk is now a dep of the user's project.

### Idempotency / re-entry

If the user already depends on mutalisk (unlikely but possible), `add_mutalisk_dep/1` skips re-adding. If the user's `:compilers` already includes `:mut_oracle`, `maybe_prepend_oracle_compiler/1` skips re-prepending (low-risk guard; user code shouldn't reference mutalisk's compiler atom).

## Module API surfaces

### `Mut.Bootstrap.Overlay` (host-side overlay generator)

Module lives in the host mutalisk checkout. Only ever called from the host BEAM — never from inside the working copy.

```elixir
defmodule Mut.Bootstrap.Overlay do
  @moduledoc "Generates the working-copy mix.exs overlay."

  @type role :: :oracle | :schema

  @spec materialize(work_copy_path :: Path.t(), role) :: :ok | {:error, term}
  def materialize(work_copy, role)

  @spec render(role) :: String.t()
  def render(role)

  @spec assert_not_umbrella!(work_copy_path :: Path.t()) :: :ok
  def assert_not_umbrella!(work_copy)
end
```

`materialize/2`:
1. Asserts `mix.exs` exists and `mix_user.exs` does not (idempotency check).
2. Calls `assert_not_umbrella!/1`; raises if umbrella detected.
3. Renames `mix.exs` → `mix_user.exs`.
4. Writes `render(role)` to `mix.exs`.

`render/1` returns the template above as a string, with the role-conditional logic baked in but the env-var lookups left dynamic so workers can change role without re-rendering.

### `Mut.Bootstrap` (in-overlay entry points)

This module lives in the host mutalisk checkout but is loaded into the **child** BEAM via the path-dep mechanism. The overlay does not call it — it's referenced indirectly via `Mix.Tasks.Compile.MutOracle` (a Mix compiler module) and `Mut.Application` (an OTP application start callback).

```elixir
defmodule Mut.Bootstrap do
  @moduledoc "Bootstrap helpers loaded into child Mix BEAMs."

  @spec role() :: :oracle | :schema | :worker | :fallback | nil
  def role do
    case System.get_env("MUTALISK_ROLE") do
      nil -> nil
      str -> String.to_existing_atom(str)
    end
  end
end
```

Tiny in v1; expand only if M2/M7 reveal a need.

### `Mix.Tasks.Compile.MutOracle` (Mix compiler module)

Module name follows Mix's `Mix.Tasks.Compile.<Camelize(atom)>` convention so the `:mut_oracle` atom in the overlay's `:compilers` list resolves to it.

```elixir
defmodule Mix.Tasks.Compile.MutOracle do
  @moduledoc "Mix compiler that registers Mut.Trace before :elixir runs."
  use Mix.Task.Compiler

  @impl true
  @spec run([String.t()]) :: {:noop, [Mix.Task.Compiler.Diagnostic.t()]}
  def run(_argv) do
    existing = Code.get_compiler_option(:tracers) || []
    Code.put_compiler_option(:tracers, [Mut.Trace | existing])

    {:ok, _pid} = Mut.Trace.Writer.start_link(jsonl_path: jsonl_path())

    System.at_exit(fn _exit_code ->
      Mut.Trace.Writer.close_with_count()
    end)

    {:noop, []}
  end

  @impl true
  @spec clean() :: :ok
  def clean, do: :ok

  @impl true
  @spec manifests() :: [Path.t()]
  def manifests, do: []

  defp jsonl_path do
    Path.join(Mix.Project.build_path(), ".mut_oracle.jsonl")
  end
end
```

Lifecycle:

- `run/1` fires **once per Mix compile cycle, before `:elixir`** because the `:mut_oracle` atom is prepended to `:compilers` in the overlay.
- Tracer and writer remain registered for the whole cycle in the same BEAM as the actual elixir compilation.
- `System.at_exit/1` registers a callback that fires when the child BEAM is shutting down, after Mix has finished all compile work and returned from `mix compile`. The sentinel write happens there. **A pre-`:elixir` compiler cannot write the sentinel synchronously after `:elixir` finishes** — that's why we use `at_exit` rather than trying to do it from `run/1`.
- If the child BEAM crashes, `at_exit` may not fire and the sentinel will be missing. The host's JSONL loader treats sentinel absence as a failure, which correctly surfaces "oracle build crashed" rather than producing a partially-loaded oracle.

### `Mut.Application` (OTP application start)

Mutalisk's own `mix.exs` declares:

```elixir
def application do
  [
    extra_applications: [:logger],
    mod: {Mut.Application, []}
  ]
end
```

So whenever any Mix project depends on mutalisk and starts its applications (which `mix test` does), `Mut.Application.start/2` fires:

```elixir
defmodule Mut.Application do
  @moduledoc "Application start callback; reads MUT_ACTIVE into :persistent_term."
  use Application

  @impl true
  @spec start(Application.start_type(), term) :: {:ok, pid}
  def start(_type, _args) do
    active =
      case System.get_env("MUT_ACTIVE") do
        nil -> 0
        "" -> 0
        str ->
          case Integer.parse(str) do
            {n, ""} when n >= 0 -> n
            _ -> 0
          end
      end

    Mut.Runtime.set_active(active)
    Supervisor.start_link([], strategy: :one_for_one, name: __MODULE__.Sup)
  end
end
```

The return value `{:ok, supervisor_pid}` from `Supervisor.start_link/2` is the only correct shape for an Application callback. **Returning `{:ok, self()}` (where `self()` is the calling process, not a started supervisor) is forbidden** — the OTP Application controller monitors the returned pid and treats its exit as the application crashing. A childless supervisor is the cheapest valid pid.

OTP boot ordering: mutalisk is a dep of the working copy, so OTP starts `:mutalisk` before `:user_app`. By the time the user's `application:start/2` runs (and any test code executes), `:persistent_term.get(Mut.Runtime.active_key())` returns the right value. **The overlay never replaces or wraps the user's `:mod` callback.** This is the design choice that protects projects that already have an Application callback — they keep theirs, mutalisk runs alongside as a separate OTP application.

### `Mut.Runtime` (already in M0 stubs)

```elixir
defmodule Mut.Runtime do
  @moduledoc "Runtime mutant selector; thin :persistent_term wrapper."

  @spec active_key() :: term
  def active_key, do: {__MODULE__, :active_mutant}

  @spec set_active(non_neg_integer) :: :ok
  def set_active(id) when is_integer(id) and id >= 0,
    do: :persistent_term.put(active_key(), id)

  @spec get_active() :: non_neg_integer
  def get_active, do: :persistent_term.get(active_key(), 0)

  @spec clear() :: :ok
  def clear, do: :persistent_term.erase(active_key()) |> then(fn _ -> :ok end)
end
```

## Failure modes & mitigations

| Scenario | Mitigation |
|---|---|
| User's `mix.exs` defines a non-MixProject module first | `Mix.Project.get!()` returns whatever was pushed by `use Mix.Project`. If user has multiple projects in one file, we capture the first; document this and fail if `mix_user.exs` defines no MixProject. |
| User's project is an umbrella | `assert_not_umbrella!/1` raises before overlay write. |
| User already depends on mutalisk | `add_mutalisk_dep/1` is idempotent. |
| `MUTALISK_PATH` not absolute | `materialize/2` calls `Path.expand/1` and validates the path contains a real mutalisk `mix.exs` before writing the overlay. |
| Tracers already configured by user | Overlay prepends, preserving existing tracers in the list. |
| `application/0` not exported by user | Overlay returns `[]` (Mix's default). |
| OTP boot order broken by `:included_applications` | We don't use `:included_applications` for mutalisk; the path-dep route is sufficient. Document as a known limitation if user has unusual app supervision needs. |
| Working copy left over from a previous run | `Mut.WorkCopy.materialize/2` accepts `force: true` to wipe and re-create; CLI defaults to fresh. |

## Verification points (M0 stubs to land)

M0 ships compile-clean stubs. Each module is created at M0; logic-bearing ones are *working* at M0, design-only ones *raise* with a marker until their milestone wires them.

| Stub | Lands in | State at M0 |
|---|---|---|
| `Mut.Runtime.{active_key, set_active, get_active, clear}` | M0 | working |
| `Mut.Application.start/2` | M0 | working (reads env, sets persistent_term, starts childless supervisor) |
| `Mut.Bootstrap.role/0` | M0 | working |
| `Mut.Bootstrap.Overlay.{materialize, render, assert_not_umbrella!}` | M0 → M2 | raises `"not yet implemented (M2)"` |
| `Mix.Tasks.Compile.MutOracle.{run, clean, manifests}` | M0 → M2 | raises (M2) |
| `Mut.WorkCopy.materialize/2` | M0 → M2 | raises (M2) |
| `Mut.Trace.Writer.start_link/1` | M0 → M2 | raises (M2) |
| `Mix.Tasks.Mut.Recompile.run/1` | M0 → M10 | raises (M10) |

The M0 unit suite asserts:

1. `Mut.Runtime` round-trips `set_active` / `get_active`.
2. `Mut.Application.start/2` honors `MUT_ACTIVE=42`, falls back to 0 when unset, empty, or garbage. Returns `{:ok, pid}` where `pid != self()` and `Process.alive?(pid)` is true (i.e., a real supervisor).
3. Every "raises" stub raises a `RuntimeError` whose message starts with `"not yet implemented"`.
4. Mutalisk's own `mix.exs` declares `mod: {Mut.Application, []}`. Asserted via `Mix.Project.config()[:application][:mod]`.

## Overlay edge-case test plan (lands with M2)

The overlay must survive each of these user-project shapes. M2 introduces a fixture-of-fixtures under `test/fixtures/overlay_cases/<case>/` for unit tests of `Mut.Bootstrap.Overlay`:

| Case | What it tests |
|---|---|
| `vanilla` | Minimal `mix.exs`, no aliases, no custom compilers, no `:mod`. Overlay should produce a working build. |
| `with_mod` | User has `application: [mod: {UserApp, []}]`. Overlay must NOT touch this. After overlay, `Mix.Project.config()[:application][:mod]` is still `{UserApp, []}`. Both `Mut.Application.start/2` and `UserApp.start/2` fire (mutalisk first, user second) per OTP dep ordering. |
| `with_aliases` | User has `aliases: ["test.all": ...]`. Overlay preserves them; `mix test.all` works in the overlay. |
| `with_compilers` | User has `compilers: [:my_compiler] ++ Mix.compilers()`. In oracle role, overlay produces `[:mut_oracle, :my_compiler] ++ Mix.compilers()`. In schema role, overlay leaves user's compilers untouched. |
| `with_deps` | User has 1+ deps. Overlay appends `{:mutalisk, path: ..., only: [:test], runtime: true}` without disturbing user deps. |
| `already_has_mutalisk` | User already lists `mutalisk` (path or hex). Overlay is idempotent — does not double-add. Logs a warning if the existing path differs from `MUTALISK_PATH`. |
| `with_elixirc_paths` | User has `elixirc_paths: ["lib", "src"]`. Overlay preserves. |
| `umbrella` | User has `apps_path: "apps"`. `Mut.Bootstrap.Overlay.assert_not_umbrella!/1` raises a clear "umbrella not supported in v1" error before any file is moved or written. |

Each overlay case has a single test that:

1. Calls `Mut.Bootstrap.Overlay.materialize/2` against a fresh COW copy of the case dir.
2. Runs `mix deps.compile mutalisk && mix compile --force` in the working copy with `MUTALISK_ROLE=oracle`.
3. Asserts compilation succeeded, the original `mix_user.exs` is preserved verbatim, and `Mix.Project.config()` queried inside the working copy returns the expected merged shape (via a tiny helper `mix run -e "..."` invocation).

These tests pin the overlay contract against real-world `mix.exs` shapes before M2 is considered done.

## Open questions for M2

These are not blockers but should be resolved during M2 implementation:

1. **Custom `Mix.Project.config_files/0`.** Some user projects override this. Overlay does not propagate. If real-world breakage shows up, add an explicit delegation.
2. **`@dialyzer` and other module attributes that affect compile.** Inherited automatically since we delegate `project/0`; no expected issues.
3. **`erlc` / Erlang sources in user project.** Compiled by the regular `:elixir` compiler list, untouched by overlay. Tracer only sees Elixir compile events. Document that Erlang source mutations are not in scope.
4. **`config/runtime.exs` reading env vars we set (`MIX_ENV`, etc.).** Standard; should just work. Confirm with a fixture test that uses a `runtime.exs`.
5. **Symlinked `deps/` becoming stale during a long run.** Workers share `deps/` symlink. If the host's `deps/` changes mid-run (`mix deps.update`), workers see it. Document as a precondition: don't run `mix deps.*` against the user project while a mutation run is in flight.

## Out of scope for v1

- Umbrella projects (`apps_path`).
- Projects without a `mix.exs` (rebar3, plain `:erlang`).
- User projects that require shell-based pre-build steps (e.g., NIFs requiring `make` before `mix compile`). M14 may add a hook; v1 assumes `mix compile` is sufficient.
- Live overlay reload (changing role without re-rendering).
- Cross-platform sandbox semantics for Windows. macOS + Linux only.
